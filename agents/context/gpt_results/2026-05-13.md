## [CLAUDE_RESULT id=T-TOSS-001] — Toss Payments 빌링 API 스펙

### 단계별 엔드포인트

| 단계 | 엔드포인트 | 필수 파라미터 | 응답 핵심 필드 |
|------|-----------|---------------|----------------|
| 1. 빌링키 발급 | `POST /v1/billing/authorizations/card` | `customerKey`, `cardNumber`, `cardExpirationYear`, `cardExpirationMonth`, `customerIdentityNumber`, `cardPassword` | `billingKey`, `customerKey`, `card`, 카드사/승인정보 |
| 2. 첫 결제 | `POST /v1/billing/{billingKey}` | Path `billingKey`, Body `customerKey`, `amount=90000`, `orderId`, `orderName` | `paymentKey`, `orderId`, `status=DONE`, `totalAmount`, `approvedAt`, `card` |
| 3. 정기 결제 | `POST /v1/billing/{billingKey}` | 동일 (서버 cron/스케줄러로 매월 트리거) | 동일 Payment 객체 |
| 4. 결제 취소 | `POST /v1/payments/{paymentKey}/cancel` | Path `paymentKey`, Body `cancelReason`; optional `cancelAmount` (부분취소) | `paymentKey`, `status`, `cancels[]`, `canceledAt`, `cancelAmount` |
| 5. 웹훅 | 콘솔에 등록한 URL | Toss가 전송 | `eventType`, `createdAt`, `data.paymentKey`, `data.status` |

### 인증/헤더

- `Authorization: Basic base64(secretKey + ":")`
- `Content-Type: application/json`
- 테스트 키: `test_sk...` 또는 `test_gsk...`
- 라이브 키: `live_sk...` 또는 `live_gsk...`
- 헤더 구조 동일, 키 값과 MID만 다름

### 핵심 주의사항

- `customerKey`: 추측 불가능한 UUID/문자열. **이메일·전화·증분 ID 금지**
- 빌링키는 발급 후 재조회 불가 → 발급 즉시 암호화 저장
- 빌링키 유효기간 = 카드 만료일
- 구독 취소는 스케줄러에서 결제 API 호출을 중단하는 방식
- **Toss는 정기결제 자체 스케줄링 제공 안 함** → 자체 cron 필요 (Vercel Cron 권장)

### 웹훅

- 핵심 이벤트: `PAYMENT_STATUS_CHANGED`
- status 값: `DONE | CANCELED | PARTIAL_CANCELED | ABORTED | EXPIRED`
- 헤더:
  - `tosspayments-webhook-transmission-time`
  - `tosspayments-webhook-transmission-retried-count`
  - `tosspayments-webhook-transmission-id`
  - `tosspayments-webhook-signature` (정산/판매자 이벤트에 주로 사용, 결제상태 웹훅엔 항상 있지 않음)
- 서명 검증: `{WEBHOOK_PAYLOAD}:{transmission-time}` → HMAC-SHA256 → `v1:` 접두 후 Base64 비교
- **결제 웹훅은 서명 미보장** → IP allowlist + 멱등키(`paymentKey` unique) + 로깅으로 방어

### LCD PRO 적용 시 이슈

- 월 90,000원 정기결제는 백엔드 cron 필수
- **라이브 정기결제는 Toss MID 별도 계약 승인 필요** ← 운영 전 체크

근거: https://docs.tosspayments.com/guides/v2/billing/integration-api · https://docs.tosspayments.com/reference/using-api/webhook-events

## [CLAUDE_RESULT id=T-KAKAO-001] — 카카오 알림톡 템플릿 등록 절차

### 단계 체크리스트

1. 카카오 비즈니스 계정 또는 발신사 콘솔 가입
2. 사업자등록 및 채널 검증
3. 알림톡 발신프로필 생성
4. 발신프로필 키 환경변수 설정
5. 템플릿 분리 등록
   - 고객용: 견적 접수 알림
   - 관리자용: 신규 견적 알림
6. 모든 변수에 예시값 입력
7. 검수 제출 → 승인 대기 → 운영 적용 → 라이브 발송 테스트

### 빈번한 검수 거절 사유

- 정보성 템플릿에 광고·마케팅 문구
- 변수 문법 오류 (`#{변수명}` 외 형식)
- 변수 40개 초과
- 본문이 변수로만 구성
- 버튼명/프리뷰에 변수 사용
- 변수 예시값 누락
- 표준화되지 않은 자유 문구
- 광고·쿠폰·이벤트·설치유도 문구

### 변수 규칙/제한

- 변수 형식: `#{variableName}`
- 변수 ≤ 40개
- 렌더 후 메시지 ≤ 약 1,000자
- 프리뷰 텍스트 ≤ 40자
- 프리뷰에 변수 사용 불가

### 알림톡 vs 친구톡

| 구분 | 알림톡 | 친구톡 |
|------|--------|--------|
| 목적 | 정보성·거래성 | 마케팅·프로모션 |
| 수신자 | 채널 친구 아니어도 수신 가능 | 보통 채널 친구 |
| 검수 | 사전 승인 필수 | 검수 부담 낮음 |
| LCD PRO 용도 | 견적 접수/관리자 알림 | 프로모션·이벤트 |
| 단가 | 정보성 단가 | 마케팅 단가 |

### LCD PRO 템플릿 권장

**고객용:**
> `#{customerName}`님, LCD PRO 견적 요청이 접수되었습니다. 요청번호: `#{requestId}`. 담당자가 곧 연락드릴 예정입니다.

**관리자용:**
> 신규 견적 접수: `#{customerName}` / `#{phoneNumber}` / `#{businessType}` / `#{requestId}`

### 주의

- 관리자 알림 템플릿은 발신사에 따라 검수 해석 다를 수 있음
- 모든 메시지를 엄격히 정보성으로, 마케팅 문구 금지
